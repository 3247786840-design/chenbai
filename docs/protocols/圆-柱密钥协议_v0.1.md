## 1. 核心决策（v0.1 + D6 草案）

### D1 — 蒸馏柱：动静态二象

- **动态扩维**：圆尚未闭合、数据类型仍在生长时，柱维可增。
- **静态刻度**：某一数据类型在某一圆内**定型封存**（约定为只读、不再改写该维语义）后，该维对该圆为**静态**；同类数据在该圆内高度标尺统一，保证可比性。

### D2 — 需求密钥：整体性

- 需求密钥是**整体对象**（一把完整的钥匙），**不因比对而分解**。
- 比对过程视为**投影 / 匹配 / 相似度运算**，而非把密钥锯碎后逐段丢失语义。

### D3 — 重合度与调取（v0 默认）

- 在**维度可对齐**的前提下，使用**加权点积类**相似度（v0：权重未学习时可为**均匀权重**或**按数据类型一张手工表**）。
- **高重合**：优先调取；**低重合**：作备选。
- 若候选**普遍低耦合**：仍取**相对最高重合**的一组，避免空转；必要时定义「全体低于阈值则不走圆、只答表层」的兜底（实现阶段再定阈值）。

### D4 — 历史参与比例

- 历史**低占比、非零**：密钥构造以**当前输入与当前观测态**为主，历史为先验；具体比例**无先验最优解**，由**本地配置**决定（建议初值区间 **10%～20%**，可观测后调参）。
- 比例需**可配置、可记录**，便于回放与防漂移审计。

### D5 — 可观测与并行

- 每种数据类型在圆上形成的**柱/特征**应能进入**观测链**（与现有 `observe-events` / 线性记忆链相容），使「管理员」可知存在哪些维、大致强度。
- **多圆并行比对**门控：用密钥与圆向量筛选**候选圆**，避免对每个圆做全量逐条文本比对。

### D6 — 管理员情感调制（草案 · 与「情感贯穿一切」对齐）

- **立场**：管理员（需求密钥侧）不是纯路由表；**同一文本需求**在不同情感态下，可对**加权系数、备选圆是否仍读、并行候选数量上限**等产生合法差异——与项目既有「感情模块贯穿一切」一致。
- **情感主源**：以 **`EmotionCore`（或当轮会话情感快照）** 为权威输入，避免与自由文本臆测混用两套心情。
- **建议可调项（实现时择一或组合，须可记录）**：
  - 对柱维权重或重合阈值做**仿射/单调**调制（先保守，避免不可解释跳变）；
  - 低重合备选圆的**是否读取**、**读取顺序**；
  - 单次并行比对**圆数量上限**（激动/高唤醒时可收窄，等）。
- **可观测**：在观测链或决策账本上保留**可追溯片段**（可复用既有 `emotionInfluence` 或新增 `adminAffectTag` 类字段），使回放能回答「当时为何更偏某圆/某备选」。

## 2. 与前例的关系（简述）

- **相似思想族**：稠密检索 / RAG、门控路由、注意力 Q–K 匹配等；本协议的特色在于与 **圆叙事、观测链、身份连续性** 的工程绑定，而非单点算法首发。

---

## 3. 实现缺口（留待 ADR / 任务）

- **工程决策与分期**：见 `docs/adr/0001-round-column-key-protocol.md`（v0 骨架：存储键名、需求指纹、`EmotionCore`→门控参数、落地阶段表）。
- 蒸馏柱向量与 `Circle` 内存储的**精确映射**与序列化格式。
- 需求密钥从用户输入生成的**可复现编码**（规则或模型，二选一或混合）。
- 加权向量 **v1** 是否从日志统计学习。
- 与现有「圆优先抽样 / 目标仲裁」的**先后关系**（前置门控 vs 替换）。
- **D6**：情感调制具体公式、与 `EmotionCore` 字段映射、默认曲线（待 ADR）。
